# L2 - Detailed Requirements

## L2-001: Create Article
**Traces to:** L1-001

An authenticated user shall be able to create a new article by providing a title, sanitized HTML body, abstract, and optional featured image. The system shall generate a URL-safe slug from the title. The article shall default to unpublished (draft) status.

**Acceptance Criteria:**
1. Given an authenticated user, when they submit a valid article with title, body, and abstract, then the article shall be persisted with a generated slug and `published = false`.
2. Given an authenticated user, when they submit an article with a title that generates a slug already in use, then the system shall return a 409 Conflict response.
3. Given an authenticated user, when they submit an article missing the title or body, then the system shall return a 400 Bad Request with field-level validation errors.
4. Given an unauthenticated request, when they attempt to create an article, then the system shall return a 401 Unauthorized response.

---

## L2-002: Edit Article
**Traces to:** L1-001

An authenticated user shall be able to update any field of an existing article (title, body, abstract, featured image, published status). Updating the title shall regenerate the slug.

**Acceptance Criteria:**
1. Given an authenticated user and an existing article, when they update the title, then the slug shall be regenerated from the new title.
2. Given an authenticated user, when they update an article with valid data, then all changed fields shall be persisted and unchanged fields shall remain intact.
3. Given an authenticated user, when they update an article with an empty title, then the system shall return a 400 Bad Request.
4. Given an authenticated user, when they update a non-existent article ID, then the system shall return a 404 Not Found.

---

## L2-003: Publish and Unpublish Article
**Traces to:** L1-001

An authenticated user shall be able to publish a draft article (setting `published = true` and recording the publication date) and unpublish a published article (setting `published = false`). Unpublishing shall preserve the original publication date.

**Acceptance Criteria:**
1. Given an authenticated user and a draft article, when they publish it, then `published` shall be `true` and `datePublished` shall be set to the current UTC timestamp.
2. Given an authenticated user and a published article, when they unpublish it, then `published` shall be `false` and `datePublished` shall retain its original value.
3. Given an unauthenticated request, when they attempt to publish an article, then the system shall return 401 Unauthorized.

---

## L2-004: Delete Article
**Traces to:** L1-001

An authenticated user shall be able to permanently delete an article. Deletion shall remove the article and disassociate any linked digital assets.

**Acceptance Criteria:**
1. Given an authenticated user and an existing article, when they delete it, then the article shall no longer be retrievable via API or public site.
2. Given an authenticated user, when they delete a non-existent article, then the system shall return 404 Not Found.
3. Given an unauthenticated request, when they attempt to delete an article, then the system shall return 401 Unauthorized.

---

## L2-005: List Published Articles (Public)
**Traces to:** L1-002

The public site shall display a paginated list of published articles, ordered by publication date descending. Each list entry shall show the title, abstract, featured image, publication date, and a link to the full article.

**Acceptance Criteria:**
1. Given published articles exist, when an anonymous user visits the articles listing page, then they shall see articles ordered by `datePublished` descending.
2. Given more articles exist than fit on one page, when the user navigates to page 2, then they shall see the next set of articles without duplicates.
3. Given no published articles exist, when the user visits the listing page, then they shall see an empty state message.
4. Given a mix of published and unpublished articles, when the listing page loads, then only published articles shall appear.

---

## L2-006: View Single Article (Public)
**Traces to:** L1-002

The public site shall render a single published article at a clean, human-readable URL based on the article slug (e.g., `/articles/{slug}`). The page shall display the title, body, featured image, publication date, and reading time estimate.

**Acceptance Criteria:**
1. Given a published article with slug `my-article`, when an anonymous user navigates to `/articles/my-article`, then the full article shall render with title, body, featured image, date, and estimated reading time.
2. Given an unpublished article, when an anonymous user navigates to its slug URL, then the system shall return a 404 page.
3. Given a non-existent slug, when a user navigates to `/articles/nonexistent`, then the system shall return a 404 page.

---

## L2-007: Semantic HTML Article Rendering
**Traces to:** L1-003

Every article page shall use semantic HTML5 elements: `<article>`, `<header>`, `<main>`, `<nav>`, `<time>`, `<figure>`, `<figcaption>`, and appropriate heading hierarchy (`<h1>` for article title, `<h2>`–`<h6>` for sub-sections). No heading levels shall be skipped.

**Acceptance Criteria:**
1. Given a published article page, when the HTML is inspected, then the article content shall be wrapped in an `<article>` element.
2. Given a published article page, when the HTML is inspected, then the article title shall be the only `<h1>` on the page.
3. Given a published article page, when the HTML is inspected, then heading levels shall be sequential (no `<h1>` followed by `<h3>` without `<h2>`).
4. Given a published article page with images, when the HTML is inspected, then images shall be wrapped in `<figure>` elements with `<figcaption>` where alt text is provided.

---

## L2-008: Structured Data (JSON-LD)
**Traces to:** L1-003

Every article page shall emit JSON-LD structured data conforming to Schema.org `Article` type, including `headline`, `datePublished`, `dateModified`, `author`, `description`, `image`, `publisher`, and `mainEntityOfPage`. The listing page shall emit `Blog` type structured data.

**Acceptance Criteria:**
1. Given a published article page, when the page source is inspected, then a `<script type="application/ld+json">` block shall contain a valid Schema.org `Article` object with all required fields populated.
2. Given a published article page, when the JSON-LD is validated against Google's Rich Results Test, then zero errors shall be reported.
3. Given the articles listing page, when the page source is inspected, then a JSON-LD block shall contain a valid Schema.org `Blog` object.

---

## L2-009: Open Graph and Twitter Card Meta Tags
**Traces to:** L1-003

Every public page shall emit Open Graph (`og:title`, `og:description`, `og:image`, `og:url`, `og:type`, `og:site_name`) and Twitter Card (`twitter:card`, `twitter:title`, `twitter:description`, `twitter:image`) meta tags with values derived from the article or page content.

**Acceptance Criteria:**
1. Given a published article page, when the HTML `<head>` is inspected, then all six `og:` tags and all four `twitter:` tags shall be present with non-empty values.
2. Given the articles listing page, when the HTML `<head>` is inspected, then `og:type` shall be `website` and `og:title` shall reflect the site name.
3. Given a published article with a featured image, when the meta tags are inspected, then `og:image` and `twitter:image` shall reference the featured image URL as an absolute URL.

---

## L2-010: Canonical URLs
**Traces to:** L1-003

Every public page shall include a `<link rel="canonical" href="...">` tag in the `<head>` with the full absolute URL of the current page. Canonical URLs shall be lowercase and shall not include trailing slashes or query parameters used for pagination.

**Acceptance Criteria:**
1. Given a published article at `/articles/my-article`, when the HTML `<head>` is inspected, then `<link rel="canonical">` shall contain `https://{domain}/articles/my-article`.
2. Given the listing page at page 2 (`/articles?page=2`), when the HTML `<head>` is inspected, then the canonical URL shall be `https://{domain}/articles` (without the query parameter).

---

## L2-011: XML Sitemap
**Traces to:** L1-003

The system shall generate and serve a dynamic XML sitemap at `/sitemap.xml` that includes all published article URLs with `<lastmod>`, `<changefreq>`, and `<priority>` elements. The sitemap shall update automatically when articles are published, updated, or unpublished.

**Acceptance Criteria:**
1. Given published articles exist, when a crawler requests `/sitemap.xml`, then the response shall be valid XML conforming to the Sitemaps protocol with one `<url>` entry per published article.
2. Given a new article is published, when `/sitemap.xml` is next requested, then the new article URL shall appear in the sitemap.
3. Given an article is unpublished, when `/sitemap.xml` is next requested, then that article URL shall no longer appear in the sitemap.
4. Given the sitemap, when inspected, then each `<url>` shall include `<loc>`, `<lastmod>`, `<changefreq>`, and `<priority>`.

---

## L2-012: Meta Title and Description
**Traces to:** L1-003

Every public page shall have a unique `<title>` tag and `<meta name="description">` tag. Article pages shall use the article title and abstract respectively. Titles shall follow the pattern `{Article Title} | {Site Name}` and shall not exceed 60 characters for the article title portion. Descriptions shall not exceed 160 characters.

**Acceptance Criteria:**
1. Given a published article with title "My Article" and abstract "A short summary", when the page loads, then `<title>` shall be "My Article | {Site Name}" and `<meta name="description">` shall contain the abstract.
2. Given an article whose title exceeds 60 characters, when the page loads, then the `<title>` shall truncate the article title portion at 60 characters with an ellipsis.
3. Given an article whose abstract exceeds 160 characters, when the page loads, then the `<meta name="description">` shall truncate at 160 characters.

---

## L2-013: robots.txt
**Traces to:** L1-004

The system shall serve a `robots.txt` file at the site root that allows all crawlers access to public pages, disallows access to administrative endpoints, and includes a `Sitemap:` directive pointing to `/sitemap.xml`.

**Acceptance Criteria:**
1. Given a crawler requests `/robots.txt`, then the response shall contain `User-agent: *` with `Allow: /` and `Disallow: /api/` (or equivalent admin paths).
2. Given a crawler requests `/robots.txt`, then the response shall contain a `Sitemap: https://{domain}/sitemap.xml` directive.

---

## L2-014: RSS/Atom Feed
**Traces to:** L1-004

The system shall serve an RSS 2.0 feed at `/feed.xml` (or `/rss.xml`) and an Atom feed at `/atom.xml` containing the most recent published articles (at minimum the 20 most recent). Each entry shall include the title, link, publication date, abstract, and author.

**Acceptance Criteria:**
1. Given published articles exist, when a feed reader requests `/feed.xml`, then the response shall be valid RSS 2.0 XML with entries for published articles.
2. Given published articles exist, when a feed reader requests `/atom.xml`, then the response shall be valid Atom XML with entries for published articles.
3. Given a new article is published, when the feed is next requested, then the new article shall appear as the first entry.
4. Given the feed, when inspected, then each entry shall include `<title>`, `<link>`, `<pubDate>`/`<updated>`, `<description>`/`<summary>`, and `<author>`.

---

## L2-015: Clean URL Structure
**Traces to:** L1-004

All public URLs shall be clean, human-readable, and lowercase. Articles shall be served at `/articles/{slug}`. The listing page shall be at `/articles`. No URL shall contain file extensions, IDs, or query parameters for content identification.

**Acceptance Criteria:**
1. Given a published article with slug `clean-architecture-guide`, when a user visits `/articles/clean-architecture-guide`, then the article shall render correctly.
2. Given a request to `/articles/Clean-Architecture-Guide` (mixed case), when processed, then the system shall 301 redirect to the lowercase canonical URL.
3. Given a request to `/articles/123` (numeric ID), when processed, then the system shall return 404 (articles are addressed by slug, not ID).

---

## L2-016: Structured Data for Agents (llms.txt)
**Traces to:** L1-004

The system shall serve an `llms.txt` file at the site root that provides a machine-readable summary of the site's purpose, content structure, and key URLs, following the emerging llms.txt convention for AI agent discoverability.

**Acceptance Criteria:**
1. Given an AI agent requests `/llms.txt`, then the response shall be a plain-text file describing the site's purpose, the content types available, and links to the sitemap and feeds.
2. Given the `llms.txt` file, when inspected, then it shall include the site name, a brief description, and URLs for `/sitemap.xml`, `/feed.xml`, and `/atom.xml`.

---

## L2-017: Server-Side Rendering with Minimal JavaScript
**Traces to:** L1-005

The public site shall be server-rendered (SSR) or statically generated (SSG). The initial HTML response shall contain the complete, readable article content with zero JavaScript required for content display. JavaScript shall only be used for optional progressive enhancements.

**Acceptance Criteria:**
1. Given a published article, when the page is requested with JavaScript disabled in the browser, then the full article content shall be visible and readable.
2. Given a published article page, when the response is inspected, then the HTML body shall contain the complete article text (not a loading skeleton or empty div waiting for JS hydration).
3. Given the public site, when the total JavaScript bundle size is measured, then it shall not exceed 50 KB gzipped.

---

## L2-018: Time to First Byte Optimization
**Traces to:** L1-005

The server shall respond to public page requests with a Time to First Byte (TTFB) of under 200ms at the 95th percentile under normal load. This shall be achieved through efficient database queries, response caching, and minimal middleware overhead.

**Acceptance Criteria:**
1. Given a published article page, when requested under normal load, then the TTFB shall be under 200ms at the 95th percentile.
2. Given the articles listing page, when requested under normal load, then the TTFB shall be under 200ms at the 95th percentile.
3. Given the server is under load (100 concurrent requests), when article pages are requested, then the TTFB shall remain under 500ms at the 95th percentile.

---

## L2-019: HTTP Caching Headers
**Traces to:** L1-005

The server shall emit appropriate `Cache-Control`, `ETag`, and `Last-Modified` headers on all public responses. Static assets (CSS, JS, images) shall use immutable cache headers with content-hashed filenames. HTML pages shall use short-lived cache with `stale-while-revalidate`.

**Acceptance Criteria:**
1. Given a static asset request (CSS/JS), when the response headers are inspected, then `Cache-Control` shall include `max-age=31536000, immutable` and the filename shall contain a content hash.
2. Given an article page request, when the response headers are inspected, then `Cache-Control` shall include `max-age=60, stale-while-revalidate=600` (or similar short-lived cache strategy).
3. Given an article page request, when the response headers are inspected, then an `ETag` header shall be present.

---

## L2-020: Image Optimization
**Traces to:** L1-005

All images served on the public site shall be optimized for web delivery. The system shall serve images in modern formats (WebP/AVIF with fallbacks), apply responsive `srcset` attributes, and enforce lazy loading for below-the-fold images.

**Acceptance Criteria:**
1. Given a published article with images, when the page HTML is inspected, then `<img>` tags shall include `srcset` and `sizes` attributes for responsive images.
2. Given a published article with images, when the page HTML is inspected, then below-the-fold images shall have `loading="lazy"` and `decoding="async"` attributes.
3. Given a featured image, when the server serves it, then the response `Content-Type` shall be `image/webp` or `image/avif` when the client's `Accept` header supports those formats.

---

## L2-021: Critical CSS Inlining
**Traces to:** L1-005

The public site shall inline critical CSS in the `<head>` of the initial HTML response to eliminate render-blocking stylesheets. Non-critical CSS shall be loaded asynchronously.

**Acceptance Criteria:**
1. Given a published article page, when the HTML `<head>` is inspected, then a `<style>` block shall contain the critical CSS for above-the-fold content.
2. Given a published article page, when the page is loaded with a slow connection (3G simulation), then the above-the-fold content shall render without waiting for external CSS files.
3. Given the public site, when Lighthouse is run, then the "Eliminate render-blocking resources" audit shall pass.

---

## L2-022: Core Web Vitals Compliance
**Traces to:** L1-005

The public site shall meet or exceed the "Good" threshold for all Core Web Vitals: Largest Contentful Paint (LCP) under 2.5s, Interaction to Next Paint (INP) under 200ms, and Cumulative Layout Shift (CLS) under 0.1.

**Acceptance Criteria:**
1. Given a published article page, when measured by Lighthouse on mobile, then LCP shall be under 2.5 seconds.
2. Given a published article page, when measured by Lighthouse on mobile, then CLS shall be under 0.1.
3. Given a published article page with interactive elements, when measured, then INP shall be under 200ms.
4. Given the articles listing page, when measured by Lighthouse on mobile, then the overall Performance score shall be 100.

---

## L2-023: JWT-Based Authentication
**Traces to:** L1-006

The back-office API shall authenticate users via JSON Web Tokens (JWT). Tokens shall be issued upon successful username/password authentication, shall expire after a configurable duration, and shall be validated on every protected endpoint.

**Acceptance Criteria:**
1. Given a valid username and password, when the user submits a login request to the authentication endpoint, then the system shall return a JWT with appropriate claims and an expiration time.
2. Given a valid JWT, when the user makes a request to a protected endpoint, then the request shall be authorized.
3. Given an expired JWT, when the user makes a request to a protected endpoint, then the system shall return 401 Unauthorized.
4. Given no JWT, when the user makes a request to a protected endpoint, then the system shall return 401 Unauthorized.

---

## L2-024: Password Security
**Traces to:** L1-006

User passwords shall be hashed using a strong, slow hashing algorithm (bcrypt, Argon2, or PBKDF2 with at least 100,000 iterations). Plaintext passwords shall never be stored or logged.

**Acceptance Criteria:**
1. Given a new user is created with a password, when the database record is inspected, then the password field shall contain a hash, not plaintext.
2. Given a login attempt with the correct password, when the system verifies it against the stored hash, then authentication shall succeed.
3. Given a login attempt with an incorrect password, when the system verifies it, then authentication shall fail with a generic error message (not revealing whether the username or password was wrong).

---

## L2-025: Input Validation and Sanitization
**Traces to:** L1-007

All user-supplied input shall be validated at the API boundary. Article body content shall be sanitized to prevent stored XSS. Input validation errors shall return 400 Bad Request with structured error details.

**Acceptance Criteria:**
1. Given an article body containing `<script>alert('xss')</script>`, when the article is saved and later rendered, then the script tag shall be stripped or escaped — it shall not execute.
2. Given a request with a title exceeding 200 characters, when submitted, then the system shall return 400 Bad Request with a validation error for the title field.
3. Given a request with SQL injection payload in any field, when submitted, then the system shall not execute the injected SQL (parameterized queries shall prevent this).

---

## L2-026: HTTPS and Security Headers
**Traces to:** L1-007

The public site and API shall be served exclusively over HTTPS. The server shall emit security headers: `Strict-Transport-Security`, `Content-Security-Policy`, `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, and `Permissions-Policy`.

**Acceptance Criteria:**
1. Given any HTTP request to the site, when processed, then the server shall redirect to HTTPS with a 301 status.
2. Given an HTTPS response, when the headers are inspected, then `Strict-Transport-Security` shall be present with `max-age` of at least 31536000.
3. Given an HTTPS response, when the headers are inspected, then `Content-Security-Policy`, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, and `Permissions-Policy` headers shall all be present.

---

## L2-027: Rate Limiting
**Traces to:** L1-007

The API shall enforce layered rate limiting on authentication endpoints (max 10 requests per minute per IP and 5 failed attempts per 15 minutes per normalized email identifier) and on write endpoints (max 60 requests per minute per authenticated user) to mitigate brute-force and abuse attacks.

**Acceptance Criteria:**
1. Given an IP address that has sent 10 login requests within one minute, when the 11th request is sent, then the system shall return 429 Too Many Requests.
2. Given a normalized email identifier with 5 failed login attempts within 15 minutes, when another failed attempt is made, then the system shall return 429 Too Many Requests or a temporary lockout response.
3. Given an authenticated user that has sent 60 write requests within one minute, when the 61st request is sent, then the system shall return 429 Too Many Requests with a `Retry-After` header.

---

## L2-028: Upload Digital Asset
**Traces to:** L1-008

An authenticated user shall be able to upload image files (JPEG, PNG, WebP, GIF, AVIF) up to 10 MB. The system shall validate file type by content inspection (not just extension), generate a unique filename, and store the asset. The asset shall be retrievable by its generated URL.

**Acceptance Criteria:**
1. Given an authenticated user, when they upload a valid 5 MB JPEG image, then the asset shall be stored and a URL shall be returned.
2. Given an authenticated user, when they upload a file with a `.jpg` extension but non-image content, then the system shall reject it with 400 Bad Request.
3. Given an authenticated user, when they upload a file exceeding 10 MB, then the system shall reject it with 413 Payload Too Large.
4. Given an unauthenticated request, when they attempt to upload, then the system shall return 401 Unauthorized.

---

## L2-029: Serve Optimized Digital Assets
**Traces to:** L1-008

Digital assets shall be served publicly with optimized delivery: content-negotiated format (WebP/AVIF when supported), long-lived cache headers, and optional query-parameter-based resizing (e.g., `?w=800`).

**Acceptance Criteria:**
1. Given a stored image and a request with `Accept: image/webp`, when the image is requested, then the response shall be in WebP format.
2. Given a stored image and a request with `?w=800`, when the image is requested, then the response shall be resized to 800px width while maintaining aspect ratio.
3. Given a stored image, when the response headers are inspected, then `Cache-Control: max-age=31536000, immutable` shall be present.

---

## L2-030: RESTful API Conventions
**Traces to:** L1-009

The API shall follow RESTful conventions: plural nouns for resource collections (`/api/articles`, `/api/digital-assets`), appropriate HTTP methods (GET for reads, POST for creates, PUT for updates, DELETE for deletes), standard status codes, and consistent JSON response envelopes.

**Acceptance Criteria:**
1. Given a GET request to `/api/articles`, when processed, then the response shall be 200 OK with a JSON array of articles.
2. Given a POST request to `/api/articles` with valid data, when processed, then the response shall be 201 Created with a `Location` header pointing to the new resource.
3. Given a DELETE request to `/api/articles/{id}` for an existing article, when processed, then the response shall be 204 No Content.
4. Given any API error, when the response is inspected, then the body shall contain a structured error object with `status`, `title`, and `detail` fields (RFC 7807 Problem Details).

---

## L2-031: API Pagination
**Traces to:** L1-009

All collection endpoints shall support cursor-based or offset-based pagination with configurable page size (default 20, max 100). Responses shall include pagination metadata (total count, current page, next/previous links).

**Acceptance Criteria:**
1. Given 50 published articles and a request to `/api/articles?pageSize=20`, when processed, then the response shall contain 20 articles and pagination metadata indicating more pages are available.
2. Given a request with `pageSize=200`, when processed, then the system shall cap the page size at 100 and return at most 100 results.
3. Given page 2 is requested, when processed, then the results shall not overlap with page 1.

---

## L2-032: Health Check Endpoint
**Traces to:** L1-010

The API shall expose a public `/health` endpoint that returns a minimal health status and a `/health/ready` endpoint that returns dependency readiness details (including database connectivity). Neither endpoint shall require authentication from deployment infrastructure.

**Acceptance Criteria:**
1. Given the application is running and the database is reachable, when `/health` is requested, then the response shall be 200 OK with `{"status": "healthy"}`.
2. Given the database is unreachable, when `/health` is requested, then the response shall be 503 Service Unavailable with `{"status": "unhealthy"}`.
3. Given the database is unreachable, when `/health/ready` is requested, then the response shall be 503 Service Unavailable with dependency detail such as `{"status": "unhealthy", "checks": {"database": "unhealthy"}}`.

---

## L2-033: Structured Logging
**Traces to:** L1-010

The application shall emit structured logs (JSON format) for all API requests (method, path, status, duration), errors (with stack traces), and significant business events (article published, user authenticated). Logs shall never contain secrets, passwords, or PII.

**Acceptance Criteria:**
1. Given any API request, when the application logs are inspected, then a structured log entry shall exist with `method`, `path`, `statusCode`, and `durationMs` fields.
2. Given an unhandled exception, when the application logs are inspected, then a structured error log entry shall exist with `exception`, `stackTrace`, and correlation ID.
3. Given a login request, when the application logs are inspected, then the log entry shall not contain the password in any field.

---

## L2-034: Database Migrations
**Traces to:** L1-011

All database schema changes shall be managed through versioned migrations using Entity Framework Core migrations. Migrations shall be forward-only in production and shall be idempotent.

**Acceptance Criteria:**
1. Given a new schema change, when a migration is generated, then it shall have a sequential, timestamped name and shall apply cleanly to a fresh database.
2. Given an existing database, when pending migrations are applied, then the schema shall be updated without data loss.
3. Given a migration has already been applied, when it is applied again, then it shall be a no-op (idempotent).

---

## L2-035: Responsive Article Listing Page
**Traces to:** L1-012

The articles listing page shall display articles in a multi-column card grid on large viewports that progressively collapses to fewer columns and eventually a single-column stack on smaller viewports. Typography and spacing shall scale proportionally.

**Acceptance Criteria:**
1. Given viewport width >= 1200px (XL), when the listing page is displayed, then articles shall render in a 3-column grid with consistent card sizes.
2. Given viewport width >= 768px and < 1200px (MD/LG), when the listing page is displayed, then articles shall render in a 2-column grid.
3. Given viewport width < 576px (XS), when the listing page is displayed, then articles shall render in a single-column stack with full-width cards.
4. Given any viewport width, when the listing page is displayed, then no horizontal scrollbar shall appear and all text shall be readable without zooming.

---

## L2-036: Responsive Article Detail Page
**Traces to:** L1-012

The article detail page shall render the article body at an optimal reading width (max ~70ch) centered on large screens, and full-width with appropriate padding on small screens. Images within articles shall scale fluidly and never overflow their container.

**Acceptance Criteria:**
1. Given viewport width >= 1200px (XL), when an article is displayed, then the body text shall be constrained to a max-width of approximately 70 characters and centered.
2. Given viewport width < 576px (XS), when an article is displayed, then the body text shall span the full viewport width with at least 16px horizontal padding.
3. Given an article with a wide image, when viewed on any viewport, then the image shall scale to fit within the container without horizontal overflow.
4. Given any viewport, when an article is displayed, then the base font size shall be at least 16px and line height at least 1.5 for readability.

---

## L2-037: Responsive Navigation
**Traces to:** L1-012

The public site shall have a navigation bar that displays as a horizontal bar on medium and larger viewports and collapses to a hamburger menu on small and extra-small viewports. Navigation shall include at minimum: home/articles link and feed link.

**Acceptance Criteria:**
1. Given viewport width >= 768px, when the navigation is displayed, then all navigation links shall be visible in a horizontal bar.
2. Given viewport width < 768px, when the navigation is displayed, then a hamburger icon shall be shown and links shall be hidden until the icon is activated.
3. Given viewport width < 768px, when the hamburger icon is activated, then the navigation links shall be revealed (slide-down, overlay, or off-canvas) and shall be tappable with at least 44x44px touch targets.

---

## L2-038: Precomputed Reading Time
**Traces to:** L1-002

Each published article shall display an estimated reading time based on word count (assuming 238 words per minute). The reading time shall be computed at save time and stored, not calculated on each request.

**Acceptance Criteria:**
1. Given an article with 1,190 words, when it is saved, then the stored reading time shall be 5 minutes.
2. Given a published article, when its detail page is rendered, then the reading time shall be displayed (e.g., "5 min read").
3. Given an article body is updated, when it is saved, then the reading time shall be recalculated.

---

## L2-039: Accessible Markup
**Traces to:** L1-002

The public site shall conform to WCAG 2.1 Level AA. All images shall have `alt` attributes, interactive elements shall be keyboard-navigable, color contrast shall meet minimum ratios (4.5:1 for body text, 3:1 for large text), and ARIA landmarks shall be used appropriately.

**Acceptance Criteria:**
1. Given any image on the public site, when the HTML is inspected, then the `<img>` element shall have a non-empty `alt` attribute (or `alt=""` for decorative images with `role="presentation"`).
2. Given the public site, when navigated using only the keyboard (Tab, Enter, Escape), then all interactive elements shall be reachable and operable.
3. Given the public site's color scheme, when contrast ratios are tested, then body text shall have at least 4.5:1 contrast and large text at least 3:1.
4. Given the public site, when run through an accessibility audit (e.g., axe-core), then zero critical or serious violations shall be reported.

## L2-040: Resource Hints and Preloading
**Traces to:** L1-005

The public site shall use `<link rel="preconnect">` for third-party origins (if any), `<link rel="preload">` for critical fonts and above-the-fold images, and `<link rel="dns-prefetch">` where applicable to minimize network latency.

**Acceptance Criteria:**
1. Given a published article page using custom fonts, when the HTML `<head>` is inspected, then `<link rel="preload" as="font">` tags shall exist for each critical font file.
2. Given a published article page with a featured image, when the HTML `<head>` is inspected, then `<link rel="preload" as="image">` shall exist for the featured image.
3. Given no third-party origins are used, when the HTML `<head>` is inspected, then no unnecessary `preconnect` tags shall be present.

## L2-041: Gzip/Brotli Compression
**Traces to:** L1-005

The server shall compress all text-based responses (HTML, CSS, JS, JSON, XML, SVG) using Brotli (preferred) or Gzip based on the client's `Accept-Encoding` header.

**Acceptance Criteria:**
1. Given a request with `Accept-Encoding: br, gzip`, when an HTML page is served, then the `Content-Encoding` response header shall be `br` (Brotli).
2. Given a request with `Accept-Encoding: gzip` only, when an HTML page is served, then the `Content-Encoding` response header shall be `gzip`.
3. Given a request with no `Accept-Encoding`, when an HTML page is served, then the response shall be uncompressed.

## L2-042: CORS Policy
**Traces to:** L1-007

The API shall enforce a strict Cross-Origin Resource Sharing (CORS) policy. Only explicitly configured origins shall be allowed. The public-facing rendered pages (same-origin) shall not require CORS. Preflight requests shall be handled correctly.

**Acceptance Criteria:**
1. Given a request from an allowed origin, when the CORS headers are inspected, then `Access-Control-Allow-Origin` shall match the requesting origin.
2. Given a request from a disallowed origin, when the CORS headers are inspected, then no `Access-Control-Allow-Origin` header shall be present.
3. Given a preflight OPTIONS request from an allowed origin, when processed, then the response shall include `Access-Control-Allow-Methods` and `Access-Control-Allow-Headers`.

## L2-043: Persistent Search Input in Global Header
**Traces to:** L1-013

The global site header shall contain a visible search input field on all public pages at all viewports. On extra-large and large viewports (>= 992px) the input shall render inline in the header as an expanded text field with a magnifying-glass icon on the left and a clear (×) button that appears only when the field has a value. On medium and smaller viewports (< 992px) the header shall show a magnifying-glass icon button; activating it shall expand the search input to full-width, overlaying or pushing down the header content, with an × button to dismiss. The input placeholder text shall read "Search articles…".

**Acceptance Criteria:**
1. Given viewport width >= 992px, when any public page is loaded, then a search text input with a magnifying-glass icon and placeholder "Search articles…" shall be visible in the header without requiring any interaction.
2. Given viewport width >= 992px and the input has a value, when the user inspects the input, then an × clear button shall be present and clicking it shall clear the input and return focus to it.
3. Given viewport width < 992px, when any public page is loaded, then only a magnifying-glass icon button shall be visible in the header (the text input shall be hidden).
4. Given viewport width < 992px and the user activates the magnifying-glass icon, when the animation completes, then a full-width search input shall be visible and focused with an × dismiss button.
5. Given viewport width < 992px and the expanded search input is visible, when the user activates the × dismiss button or presses Escape, then the search input shall collapse and focus shall return to the magnifying-glass icon.
6. Given any viewport, when the search input is visible, then its touch/click target shall be at least 44 × 44 CSS pixels.

## L2-044: Keyboard Shortcut to Focus Search
**Traces to:** L1-013

The public site shall support a keyboard shortcut that moves focus directly to the search input from anywhere on the page. Pressing the `/` key when focus is not inside a text field, textarea, or contenteditable element shall focus the search input and select any existing value. The shortcut hint shall be displayed inside the search input field on large viewports.

**Acceptance Criteria:**
1. Given focus is on the page body (not in any text field), when the user presses the `/` key, then the search input shall receive focus and any existing value shall be selected.
2. Given focus is inside a `<textarea>`, `<input>`, or `contenteditable` element, when the user presses `/`, then the shortcut shall not trigger and the keystroke shall be typed normally.
3. Given viewport width >= 992px, when the search input is inspected, then a `/` key hint shall be visually present inside the trailing area of the input (e.g., as a `<kbd>/</kbd>` badge) and shall disappear once the input receives focus or has a value.
4. Given the search input has focus, when the user presses Escape, then focus shall return to the previously focused element and the input shall be cleared on small viewports.

## L2-045: Live Autocomplete Suggestions
**Traces to:** L1-013

As the user types in the search input, the system shall display a dropdown suggestion list beneath the input after a debounce interval of 200 ms. Suggestions shall be sourced from matching published article titles and shall update on every keystroke beyond the minimum query length. The dropdown shall show a maximum of 8 suggestions. Each suggestion row shall display the article title with the matching substring visually highlighted. Selecting a suggestion shall navigate directly to that article's detail page, not to the search results page.

**Acceptance Criteria:**
1. Given the search input has focus and the user has typed 2 or more characters, when 200 ms have elapsed since the last keystroke, then a suggestions dropdown shall appear beneath the input showing up to 8 matching published article titles.
2. Given suggestions are visible, when the matching substring is inspected in the rendered HTML, then the matched characters shall be wrapped in a `<mark>` element (or equivalent highlighted element) to distinguish them from non-matching characters.
3. Given suggestions are visible, when the user presses the Down Arrow key, then focus shall move to the first suggestion; subsequent Down Arrow presses shall advance through the list; Up Arrow shall move backwards; pressing Up on the first item shall return focus to the input.
4. Given a suggestion has keyboard focus or is hovered, when the user presses Enter or clicks the suggestion, then the browser shall navigate to that article's URL (`/articles/{slug}`) without going via the search results page.
5. Given suggestions are visible, when the user presses Escape, then the dropdown shall close and focus shall remain in the search input with the typed value preserved.
6. Given the user types fewer than 2 characters, when the input is inspected, then no suggestions dropdown shall appear.
7. Given the user types a query that matches no published article titles, when the dropdown appears, then it shall not render (no empty dropdown shell shall be shown).
8. Given the suggestions dropdown is visible, when the user clicks outside the input and dropdown, then the dropdown shall close.
9. Given the suggestions API is queried, when the request is inspected, then the query parameter shall be HTML-entity-encoded and the endpoint shall return results within 100 ms at the 95th percentile.

## L2-046: Search Results Page
**Traces to:** L1-013

Submitting the search form (pressing Enter or clicking the magnifying-glass icon) shall navigate to a dedicated search results page at `/search?q={encoded-query}`. The page shall display the query echoed back in a heading ("Results for "{query}""), the total result count ("X results"), and an ordered list of result cards. Results shall be ranked by relevance (full-text match score) with recency as a tiebreaker. The page title shall follow the pattern `{query} — Search | {Site Name}`.

**Acceptance Criteria:**
1. Given the user types "clean architecture" and submits the search, when the results page loads, then the URL shall be `/search?q=clean+architecture` (or percent-encoded equivalent) and the `<title>` shall be `clean architecture — Search | {Site Name}`.
2. Given the results page loads with a non-empty query, when the page is inspected, then a heading shall contain the text `Results for "clean architecture"` and a count element shall display the total number of matching articles.
3. Given published articles match the query, when the results page loads, then articles with the query terms in the title shall rank above articles where the terms appear only in the abstract or body.
4. Given the query string is modified in the URL, when the page reloads, then the search input in the header shall be pre-populated with the current query value.
5. Given an unauthenticated user visits `/search?q=anything`, when the page loads, then a 200 OK response shall be returned with the result list — no authentication is required to search.

## L2-047: Search Result Card
**Traces to:** L1-013

Each entry on the search results page shall be rendered as a card containing: the article title as a clickable `<h2>` link to the article detail page, a short excerpt of the abstract (truncated to 160 characters with an ellipsis if longer), the publication date formatted as a human-readable string (e.g., "April 5, 2026"), a "Article" content-type label, and — if the article has a featured image — a thumbnail image on the right side of the card on medium and larger viewports. The query terms within the title and excerpt shall be visually highlighted using a `<mark>` element.

**Acceptance Criteria:**
1. Given a search result card, when inspected, then it shall contain: an `<h2>` with a link to `/articles/{slug}`, an excerpt of the abstract ≤ 160 characters, a formatted publication date, and an "Article" label.
2. Given the article title contains the search term, when the card is rendered, then the matching substring in the title shall be wrapped in `<mark>`.
3. Given the abstract contains the search term, when the card is rendered, then the matching substring in the excerpt shall be wrapped in `<mark>`.
4. Given the article has a featured image and viewport >= 768px, when the card is rendered, then the featured image shall appear as a thumbnail (aspect ratio 16:9, max-width 180px) on the right side of the card.
5. Given the article has no featured image, when the card is rendered, then no image placeholder or broken image element shall appear.
6. Given viewport width < 576px, when the card is rendered, then the thumbnail image shall not be shown and the card shall render as a single-column text-only layout.

## L2-048: Search Empty State
**Traces to:** L1-013

When a search query returns zero results, the search results page shall display a meaningful empty state that communicates clearly that no articles matched and offers guidance. The empty state shall include: an icon or illustration, the message "No results found for "{query}"", a suggestion to try different keywords, and a link back to the articles listing page.

**Acceptance Criteria:**
1. Given a query that matches no published articles, when the results page loads, then the HTTP response status shall be 200 OK and the page shall display "No results found for "{query}"".
2. Given the empty state is displayed, when the page is inspected, then it shall contain a prompt to try different keywords and a navigable link to `/articles`.
3. Given the empty state is displayed, when the page is inspected, then no result count element shall show "0 results" as the only feedback — the dedicated empty state component shall replace the result list area.
4. Given an empty query string (`/search?q=`), when the page loads, then the results page shall display the empty state rather than attempting a search.

## L2-049: Search Results Pagination
**Traces to:** L1-013

The search results page shall paginate results at 10 articles per page. Pagination shall be rendered as a row of numbered page links with Previous and Next controls, matching the visual style of the articles listing page. The current page number shall be reflected in the URL as `/search?q={query}&page={n}`. The first page shall not include a `page` parameter.

**Acceptance Criteria:**
1. Given more than 10 articles match a query, when the search results page loads, then only 10 results shall be shown and a pagination component shall appear below the results.
2. Given the pagination component is visible, when the user navigates to page 2, then the URL shall update to `/search?q={query}&page=2` and the next 10 results shall be displayed without duplicates.
3. Given the user is on page 1, when the pagination component is inspected, then the Previous control shall be present but disabled (or visually indicated as non-navigable).
4. Given the user is on the last page, when the pagination component is inspected, then the Next control shall be present but disabled.
5. Given 10 or fewer articles match the query, when the search results page loads, then no pagination component shall appear.

## L2-050: Search Responsive Layout
**Traces to:** L1-013

The search results page shall adapt its layout across all five breakpoints. On extra-large viewports (>= 1200px) the results list shall be constrained to the same max-width reading column as article detail pages. Result cards shall display thumbnails on medium and larger viewports and suppress them on small and extra-small viewports. The search input in the header shall behave according to L2-043 at each breakpoint.

**Acceptance Criteria:**
1. Given viewport width >= 1200px (XL), when the search results page is displayed, then the results list shall be centered in a constrained column (max-width approximately 860px) and no horizontal scrollbar shall appear.
2. Given viewport width >= 768px (MD/LG/XL), when a result card is displayed, then it shall render in a two-column layout with text on the left and the featured image thumbnail on the right.
3. Given viewport width < 576px (XS), when result cards are displayed, then each card shall render in a single-column, full-width layout with no thumbnail.
4. Given any viewport width, when the search results page is displayed, then no content shall overflow its container horizontally.
5. Given viewport width < 992px (SM/XS), when the search results page loads, then the search input in the header shall be in its collapsed icon-only state per L2-043.

## L2-051: Search Accessibility
**Traces to:** L1-013

The search feature shall conform to WCAG 2.1 Level AA. The search input shall have an accessible label, the suggestions dropdown shall be implemented as an ARIA `combobox` pattern (role="combobox", role="listbox", role="option"), and the results page shall use appropriate landmarks and heading hierarchy. All interactive search elements shall be operable by keyboard alone.

**Acceptance Criteria:**
1. Given the search input, when inspected with an accessibility tool, then it shall have an associated `<label>` or `aria-label` with value "Search articles".
2. Given the autocomplete suggestions dropdown is open, when the ARIA attributes are inspected, then the input shall have `role="combobox"`, `aria-expanded="true"`, and `aria-controls` pointing to the suggestions list; the list shall have `role="listbox"` and each item `role="option"`.
3. Given the search results page, when inspected, then results shall be wrapped in a `<main>` landmark, the result count shall be in a live region (`aria-live="polite"`) so screen readers announce the count after it updates, and each result card title shall be an `<h2>`.
4. Given the search results page, when navigated using only the keyboard, then all links, pagination controls, and the header search input shall be reachable via Tab in a logical order.
5. Given any search UI element, when color contrast is tested, then text and interactive controls shall meet the 4.5:1 minimum ratio for normal text and 3:1 for large text and UI components.

## L2-052: Search Performance
**Traces to:** L1-013

Search queries shall execute server-side against a full-text index on the articles table. The search results page shall respond within 300 ms TTFB at the 95th percentile under normal load. The autocomplete suggestions endpoint shall respond within 100 ms at the 95th percentile. Search shall not block or degrade the rendering of non-search pages.

**Acceptance Criteria:**
1. Given a search query submitted via the results page URL, when the server processes the request under normal load, then the TTFB shall be under 300 ms at the 95th percentile.
2. Given a live autocomplete request is issued (debounced at 200 ms), when the server processes it, then the JSON response shall be returned within 100 ms at the 95th percentile.
3. Given the database contains 10,000 published articles, when a full-text search query is executed, then the response time shall remain within the above thresholds (i.e., the full-text index shall not degrade with data volume).
4. Given a non-search page (e.g., article detail), when it is loaded while a concurrent search query is executing, then the non-search page TTFB shall not exceed 200 ms (searches shall not starve other queries).

## L2-053: Search Input Security
**Traces to:** L1-013

All search input shall be treated as untrusted user input. The query parameter shall be HTML-encoded before rendering in the results page heading and `<title>`. The search API endpoint shall enforce a maximum query length of 200 characters. The system shall not reflect raw query strings into the page in any context that could enable XSS.

**Acceptance Criteria:**
1. Given a query containing `<script>alert(1)</script>`, when the results page is rendered, then the script tag shall appear as escaped text (e.g., `&lt;script&gt;`) in the heading — it shall not execute.
2. Given a query containing `" onmouseover="alert(1)`, when the results page is rendered, then no injected event handler shall execute and the query shall render as escaped text.
3. Given a query exceeding 200 characters, when submitted to the search endpoint, then the system shall return 400 Bad Request and no search shall be performed.
4. Given the `<title>` tag on the search results page, when inspected, then the query value shall be HTML-entity-encoded so that special characters cannot break out of the tag context.
